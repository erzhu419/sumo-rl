既然你确定**“决策一管到底”**且**“奖励到站才结清”**，这说明你的模型本质上是在处理一个**长程路径规划（Long-term Trajectory Planning）**问题。

在这种场景下，SAC 表现不如硬规则（-0.87M vs -0.71M）的根本原因在于：**动作的“放大效应”与“信用分配”在稀疏决策下的不匹配。**

以下是针对这一特定机制的 5 条针对性优化方案：

### 1. 消除 Ensemble 的“过度防御” (Pessimism Reduction)
在“一站决策一次”这种极低频率的采样中，Ensemble SAC 的 $Q = \min(Q_1, ..., Q_n)$ 是性能杀手。
*   **分析：** 交通流有巨大的随机性。假设 SAC 选了一个合理的 `SpeedFactor`，但路上正好随机刷了一辆慢车导致 Reward 很低。Ensemble 中的某个 Q 网络会死记硬背住这个“错误”，导致 $\min(Q)$ 变得极度保守。


### 2. 动作空间的“动态门限”引导 (Guidance via Dynamic Action Clipping)
目前的启发式引导如果是通过 MSE Loss 做的，它只是在“建议”模型靠近硬规则。
*   **分析：** 由于 `SpeedFactor` 的微小偏差会在长距离上被放大，SAC 的随机搜索（Noise）会产生大量“垃圾轨迹”。
*   **优化建议：** 不要只在 Loss 里加引导，直接在 `act()` 层面对动作进行物理限制。
    *   **做法：** 假设硬规则计算出的 Holding 是 $H_{rule}$，SpeedFactor 是 $S_{rule}$。
    *   **限制：** 让 SAC 的输出范围限制在 $[H_{rule}-5, H_{rule}+5]$ 和 $[S_{rule}-0.1, S_{rule}+0.1]$。
    *   **理由：** 这样 SAC 的起步点就是 -0.71M，它的任务从“学习如何控制”变成了“在硬规则基础上做增量优化”。这能有效防止你提到的“在硬规则附近震荡”因为你压缩了它震荡的振幅。


### 5. 降低折扣因子 $\gamma$ (Discount Factor Adjustment)
*   **优化建议：** 尝试将 $\gamma$ 从 0.99 降低到 **0.8 甚至更低**。
*   **理由：** 在“一管到底”的逻辑下，A 站的决策对 B 站的影响是直接的，但对 C 站、D 站的影响已经被复杂的交通流扰动冲淡了。过高的 $\gamma$ 会让 Agent 试图去优化极其遥远的未来，反而把当前的站间控制搞得一团糟。**在这种模式下，RL 应该更像一个“上下文老虎机（Contextual Bandit）”。**

如果采用了 **Residual Learning (残差学习)** 架构，模型依然在 Baseline (硬规则) 附近震荡而不突破，通常不是因为算法“没学会”，而是因为**“信号被噪声淹没了”**或者**“评价体系太迟钝”**。

在 SUMO 这种高随机性的交通仿真中，以下是导致“基准线震荡”的 4 个深层原因及其对策：

### 1. 结构性尺度失配 (Structural Scale Mismatch)
这是残差学习中最常见的问题。你的最终动作 $u = \text{Action}_{base} + \text{Residual}_{rl}$。
*   **原因：** 基准动作（如 Speed Factor = 1.0）在数值量级上远大于残差动作（如 $\pm 0.05$）。对于 Critic 网络来说，动作空间的微小变化引起的 Q 值波动非常小，很容易被神经网络的函数近似误差（Function Approximation Error）覆盖掉。
*   **对策：** 
    *   **Critic 归一化 (Critic Normalization)：** 尝试对奖励进行 Scaling，或者在 Critic 网络中使用 Layer Norm，提高其对微小动作变化的敏感度。
    *   **增大残差权重：** 如果 $0.05$ 的修正根本看不出效果，尝试给残差项加一个可学习的缩放系数，或者稍微放宽限制。

### 2. 熵支配问题 (Entropy Dominance in SAC)
SAC 的核心是最大化熵（Exploration）。
*   **原因：** SAC 算法会自动调整 $\alpha$ (温度参数) 以维持一定的探索度。在残差学习中，如果 $\alpha$ 过大，SAC 产生的“随机扰动”会远大于它学到的“有效修正”。结果就是：Agent 只是在基准线周围“乱抖”，这些抖动在交通流中被视为干扰，导致性能甚至不如干净的硬规则。
*   **对策：** 
    *   **调小目标熵 (Target Entropy)：** 手动降低 SAC 的目标熵，或者给 $\alpha$ 设置一个很小的上限。
    *   **后期衰减：** 随着训练进行，强制减小探索噪声。

### 3. 信用分配的“稀疏性”与“时滞”
你提到“到站决策，到站给奖”，这意味着这是一个 **长程开环控制**。
*   **原因：** 从 A 站到 B 站，中间可能发生了无数次随机车辆换道、路口干扰。最终的 Reward（延误、排队）受到这些随机因素的影响，远大于你那个 $0.02$ 的 Speed Factor 修正带来的影响。
*   **信号增益比太低：** 就像在暴风雨的海面上投一颗小石子，波纹（SAC 的贡献）完全被浪花（交通噪声）淹没了。
*   **对策：** 
    *   **增加训练样本量：** 这种稀疏奖励需要海量数据来“磨平”随机性。
    *   **引入“对比奖励” (Relative Reward)：** 奖励不应该只是绝对延误，而应该是 `Reward = -(当前延误 - 历史同工况平均延误)`。这能抵消掉交通流本身的随机波动，让 SAC 看到自己动作的真实贡献。

### 4. Ensemble 的悲观偏置 (Pessimistic Bias)
*   **原因：** Ensemble SAC 取多个 Q 网络的最小值。在交通流中，偶尔的“运气差”（比如由于随机流量导致的 Bunching）会产生极低的 Q 值。Ensemble 会由于“悲观”而永远记住这些极端的坏情况，导致它不敢尝试任何可能带来收益的 Speed Factor 提升。
*   **对策：** 
    *   **减少 Ensemble 数量：** 在残差学习中，安全性已经由硬规则保底，不再需要极度悲观的 Ensemble。建议直接换回 **标准的 Twin-Q SAC (2个Q网络)**。

### 1. 论文核心发现：为什么你的 SAC 震荡且低效？

论文指出，残差 RL 存在两个致命瓶颈，这可能正是你代码中 `sac_ensemble` 表现差的原因：

*   **冷启动病理 (Cold Start Pathology)**：
    *   **现象**：SAC 的 Critic 网络刚开始是随机初始化的，它根本不知道“硬规则”执行得好坏。
    *   **后果**：由于你的决策是“一管到底”（站间决策），样本极其稀疏。如果 Critic 没能先理解硬规则的价值，它给出的梯度就是随机的，会导致 Actor 乱改速度，把原本稳定的交通流搞乱。
*   **结构性尺度失配 (Structural Scale Mismatch)**：
    *   **现象**：你的 `SpeedFactor` 基准通常是 1.0 左右，而 SAC 输出的残差（修正量）可能只有 0.05。
    *   **后果**：对于 Critic 网络来说，**输入 1.0 和输入 1.05 的区别太小了**。由于神经网络的特性，它会忽略掉这个微小的动作差异，导致“信用分配”失败——即 SAC 根本不知道那 -0.16M 的差距是因为哪个微调动作引起的。

### 2. 基于论文的 3 个具体优化方案（针对你的代码）

你可以直接根据论文提出的 **DAWN (Data-Anchored Warmup and Normalization)** 框架修改你的脚本：

#### 方案一：数据锚定预热 (Data-Anchored Warmup, DAW)
**不要**直接开始训练，先让 Critic “认路”。
*   **操作**：在正式启动训练循环（`update`）之前，先运行 `run_hard_rule_baseline.py` 的逻辑，收集例如 20,000 条轨迹数据存入 Replay Buffer。
*   **关键点**：这 20,000 条数据中，动作全是硬规则动作，残差为 0。
*   **效果**：这样 Critic 在第一次更新 Actor 之前，就已经通过这些“锚点数据”学会了硬规则在不同路况下的预期 Reward。它能立刻分辨出哪些修正会让 Reward 变好。

#### 方案二：Critic 网络的层归一化 (Critic Layer Normalization) —— **最核心建议**
这可能是解决你“震荡”和“梯度问题”的银弹。
*   **操作**：在 Critic 网络的**输入层之后**或**隐藏层中**加入 `nn.LayerNorm`。
*   **论文原理**：LayerNorm 可以消除输入绝对数值（Base Action = 1.0）的支配地位，放大微小变动（Residual = 0.05）的特征权重。
*   **代码修改示例**：
    ```python
    class Critic(nn.Module):
        def __init__(self):
            # 在合并 state 和 action 后的第一层加入 LayerNorm
            self.layer_norm = nn.LayerNorm(hidden_dim)
            
        def forward(self, state, action):
            x = torch.cat([state, action], dim=1)
            x = self.fc1(x)
            x = self.layer_norm(x) # 恢复对微小动作变化的敏感度
            # ... 后续层
    ```

#### 方案三：解决“熵支配”导致的价值坍塌 (Entropy Dominance)
论文第 4 页分析了为什么显式预热 Critic 会失败。
*   **分析**：在 SAC 中，如果 $\alpha$ (温度参数) 自动调整，由于你的动作在初始化时接近确定性（残差接近 0），$\log \pi$ 会变得极大。这会导致 SAC 的目标函数被“熵”统治，忽略了真实的交通奖励，导致 Q 值坍塌（见论文 Figure 5）。
*   **建议**：
    1.  **固定一个极小的 $\alpha$**（例如 0.01），不要在初期让它自动调整。
    2.  或者按照方案一，只做**数据锚定（填 Buffer）**，而不做**显式预热（不预先更新神经网络）**。

### 3. 针对你“一管到底”机制的特殊修正

既然是站间决策，动作和奖励都是稀疏的：

1.  **残差缩放 ($\lambda$)**：论文提到 $\lambda$ 设为 0.1 是很稳健的。确保你的 SAC 输出是通过 `torch.tanh(x) * 0.1` 叠加到硬规则上的。
2.  **不要 Clip 最终动作**：正如你担心的，`clip` 会杀掉梯度。如果你使用了 `base + tanh(res) * 0.1`，其结果自然就在 `[base-0.1, base+0.1]` 范围内，**完全不需要再加 clip**，这样梯度是连续的。

### 总结优化步骤：
1.  **Critic 加 LayerNorm**（解决尺度失配，让 Critic “看见” SAC 的微小修正）。
2.  **先跑 20k 步硬规则存入 Buffer**（解决冷启动，给 Critic 一个价值基准）。
3.  **Action 采用加法式残差**：$a = a_{base} + 0.1 \cdot \text{tanh}(a_{sac})$。
4.  **固定或限制 $\alpha$**：防止熵项淹没站间奖励信号。

这篇论文的核心哲学是：**RL 跑不赢硬规则，不是因为 Actor 不行，而是因为 Critic 在大尺度的基准动作面前“变瞎了”。** 加上 LayerNorm 相当于给 Critic 戴上了显微镜。
